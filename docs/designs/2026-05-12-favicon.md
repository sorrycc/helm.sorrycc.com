# Favicon — Octopus mascot

## 1. Background

The site has no favicon today. `public/index.html` `<head>` (L3-21) declares `<meta>`, `<title>`, font preloads, and a theme bootstrap script — but no `<link rel="icon">`. Browsers fall back to a blank/generic tab icon, and Chromium/Safari/RSS readers issue 404-noisy auto-requests for `/favicon.ico`.

The brand already has a well-defined octopus mascot — `#octo-mark` symbol at `public/index.html` L32-38, filled with `og-shared` linear gradient (`#5fd9c7 → #3ec9b6 → #0d6b62`, declared L29-31, stop colors from `src/styles.css` L135-137). It is used inline at 22×22 in nav (L62) and footer (L834).

Additionally, `wrangler.jsonc` L7 sets `not_found_handling: "single-page-application"`. This means missing static files at `/favicon.ico` return `index.html` (HTTP 200, `Content-Type: text/html`), not 404 — committing the binary ICO is therefore mandatory to avoid serving HTML in response to favicon requests.

## 2. Requirements Summary

Add a favicon that reuses the existing mascot, served as a transparent SVG with a multi-size ICO legacy fallback. No PWA / apple-touch / dark-mode-variant scope. No build pipeline changes; static files committed to `public/`.

## 3. Acceptance Criteria

1. `public/favicon.svg` exists, is a standalone SVG (gradient inlined, no CSS dep), uses the same path `d` from `#octo-mark` (`public/index.html` L33), preserves the two pupil circles (L34-35), and uses a tightened viewBox so the glyph fills the canvas vertically at 16×16 (within ~1px of top/bottom edges; small horizontal letterbox is acceptable). White eye-highlight dots from L36-37 are intentionally dropped (sub-pixel at 16×16 — they alias to nothing).
2. `public/favicon.ico` exists, contains 16×16, 32×32, and 48×48 frames, transparent background (PNG-frame ICO; modern browsers only — pre-Win10 not in scope), same teal-gradient octopus.
3. `public/index.html` `<head>` contains one new line: `<link rel="icon" type="image/svg+xml" href="/favicon.svg" />`, placed immediately after `<title>` (L6).
4. Loading the site shows the teal octopus in the browser tab in Chrome, Firefox, and Safari (any of SVG or ICO is acceptable per browser).
5. *(merged into AC4 — Safari ≥14 supports SVG favicons; per-browser source is implementation-defined.)*
6. `curl -sI /favicon.ico` returns HTTP 200 **and** `Content-Type` starts with `image/` (i.e. the committed binary file short-circuits Wrangler's SPA fallback to `index.html`).
7. No new `package.json` dependencies; `bun run build` behavior is unchanged.
8. `public/styles.css` (build output) and `public/fonts/` remain the only gitignored items under `public/`; both new favicon files are committed via explicit `git add public/favicon.svg public/favicon.ico` (not `git add public/`, which would stage build output).
9. Size budget: `public/favicon.svg` < 1 KB; `public/favicon.ico` < 4 KB.

## 4. Problem Analysis

- **Approach A — Use a raster-only `favicon.ico`** → rejected: doesn't scale to retina/high-DPI tabs (browser favicon area is 16px CSS, but devices render at 2–3×).
- **Approach B — SVG only, no ICO** → rejected: with Wrangler's SPA fallback, `/favicon.ico` returns HTML rather than 404, serving garbage to crawlers/RSS readers that only fetch ICO.
- **Approach C — Full PWA set** (SVG + ICO + apple-touch + manifest + 192/512 PNGs) → rejected: YAGNI for a static marketing landing page with no install/home-screen story.
- **Chosen approach — SVG + ICO** → SVG handles modern browsers at any DPI; committed ICO covers legacy + auto-requests + silences SPA fallback; minimal asset count.

## 5. Decision Log

**1. Asset set**
- Options: A) SVG only · B) SVG + ICO · C) Full PWA set
- Decision: **B)** — committing the binary ICO is mandatory because of `wrangler.jsonc` SPA fallback; full PWA set is YAGNI.

**2. Background**
- Options: A) Transparent · B) Solid dark backdrop · C) Solid teal backdrop
- Decision: **A)** — existing inline usage at index.html L62 and L834 already proves the gradient mark reads on both `bg-white/60` (nav) and `dark:bg-ink-950/70`.

**3. Dark-mode adaptation**
- Options: A) Single icon · B) Two SVGs via `media="(prefers-color-scheme: dark)"`
- Decision: **A)** — favicon represents brand identity independent of viewer theme; teal mid `#3ec9b6` reads on both light/dark browser tab chrome.

**4. Artwork variant & viewBox**
- Options: A) Exact path verbatim, original viewBox · B) Exact path + tightened viewBox · C) Redraw with tentacles
- Decision: **B)** — same path/eye identity as nav/footer. Path bbox is x ∈ [8, 24], y ∈ [4, 24.4] (≈ 16×20.4). Use `viewBox="6 3 20 22"` — 2px horizontal breathing, 1px vertical breathing on each side, glyph fills ~93% of the viewBox vertically. **Revised in Phase 4 review:** original proposal was `viewBox="6 2 20 24"`; reviewer correctly noted excess bottom whitespace (path max y ≈ 24.4, not 26). Tightened to `6 3 20 22`. Aspect ratio is 10:11 (not perfectly square) — at 16×16 the browser letterboxes ~0.75px on each side horizontally, which is visually negligible.
- **Eye highlights:** the four-circle eye structure from `#octo-mark` (`r=1.1` pupils + `r=.35` white catchlights) is reduced to two pupils for the favicon. The `r=.35` highlights are sub-pixel at 16×16 (≈ 0.5 device px) and alias to nothing; dropping them prevents speckling without changing the identity at common favicon sizes.

**5. Build integration**
- Options: A) Static committed files, no build change · B) SVG static, ICO generated at build · C) Both generated at build
- Decision: **A)** — README L13 already commits everything in `public/` except `styles.css` build output and `fonts/`. No build dep needed; ICO regen is one-time.

**6. ICO generation tool (one-time, not committed to build)**
- Options: A) `bunx png-to-ico` with `rsvg-convert` PNG renderer · B) ImageMagick · C) Custom inline Bun packer
- Decision: **C)** — **Revised during implementation:** original choice was A), but `bunx png-to-ico` (a) adds an unrequested 4th 256×256 frame, and (b) packs frames as raw BMP-with-mask (32bpp uncompressed) — the resulting ICO ballooned to ~285 KB, far over the 4 KB budget (AC9). Replaced with a ~25-line inline Bun script (`/tmp/make-ico.ts`, not committed) that packs the rsvg-rendered PNGs as PNG-format ICO frames (the modern ICO variant; decoded by all current browsers, pre-Win10 explicitly out of scope per Decision 1). Final size: 3.4 KB. `rsvg-convert` is preinstalled at `/opt/homebrew/bin/rsvg-convert`; `--background-color=none` ensures transparent PNG output. No `package.json` change.

**7. `<link>` placement**
- Options: A) After `<title>` · B) After all `<meta>` · C) After `<link rel="stylesheet">`
- Decision: **A)** — immediately after `<title>` (L6), before stylesheet/font preloads. Standard convention; doesn't disturb the preload group.

**8. Gradient inlining**
- Options: A) Inline `<defs><linearGradient>` with stop colors · B) Reference external CSS · C) Use solid fill `#3ec9b6`
- Decision: **A)** — inline `<defs><linearGradient id="fav-grad" x1="0" y1="0" x2="1" y2="1">` with stops `#5fd9c7 → #3ec9b6 → #0d6b62`, ref via `fill="url(#fav-grad)"`. Id renamed from source's `og-shared` to `fav-grad` to avoid any namespace collision if the SVG is ever inlined into the document by an extension. 3-stop gradient kept (not simplified to 2-stop) to maintain identity with nav/footer usage; verification step §8.4 must include a visual check at 16×16 to confirm the `#0d6b62` foot doesn't smudge.

### File: `public/favicon.ico`

Generated one-time from `public/favicon.svg` (Decision 6 revision). Step 1 — render PNGs via `rsvg-convert`:

```bash
rsvg-convert --background-color=none -w 16 -h 16 public/favicon.svg -o /tmp/favicon-16.png
rsvg-convert --background-color=none -w 32 -h 32 public/favicon.svg -o /tmp/favicon-32.png
rsvg-convert --background-color=none -w 48 -h 48 public/favicon.svg -o /tmp/favicon-48.png
```

Step 2 — pack into PNG-format ICO with a one-shot Bun script (not committed):

```ts
// /tmp/make-ico.ts
const pngs = await Promise.all(
  [16, 32, 48].map(async (size) => {
    const buf = await Bun.file(`/tmp/favicon-${size}.png`).arrayBuffer();
    return { size, data: new Uint8Array(buf) };
  })
);

const HEADER_SIZE = 6;
const ENTRY_SIZE = 16;
const dataOffset = HEADER_SIZE + ENTRY_SIZE * pngs.length;
const totalSize = dataOffset + pngs.reduce((s, p) => s + p.data.length, 0);

const out = new Uint8Array(totalSize);
const view = new DataView(out.buffer);

// ICONDIR
view.setUint16(0, 0, true);           // reserved
view.setUint16(2, 1, true);           // type=1 (ICO)
view.setUint16(4, pngs.length, true); // count

// ICONDIRENTRY[]
let offset = dataOffset;
pngs.forEach((png, i) => {
  const e = HEADER_SIZE + i * ENTRY_SIZE;
  out[e] = png.size === 256 ? 0 : png.size;     // width (0 = 256)
  out[e + 1] = png.size === 256 ? 0 : png.size; // height
  view.setUint16(e + 4, 1, true);               // planes
  view.setUint16(e + 6, 32, true);              // bpp
  view.setUint32(e + 8, png.data.length, true); // bytes-in-res
  view.setUint32(e + 12, offset, true);         // offset
  out.set(png.data, offset);
  offset += png.data.length;
});

await Bun.write("public/favicon.ico", out);
```

Run: `bun /tmp/make-ico.ts`. Result: 3424-byte ICO with three PNG-compressed frames (16/32/48), transparent. `rsvg-convert` is preinstalled (`/opt/homebrew/bin/rsvg-convert`); no `package.json` change.
